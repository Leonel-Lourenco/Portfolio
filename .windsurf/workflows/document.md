## Objective
Generate clear, accurate, and maintainable documentation.

## Requirements
1. **Understand Before Documenting**
   - Read the code/feature thoroughly
   - Identify the intended audience (developers, users, operators)
   - Understand the "why" not just the "what"

2. **Structure Appropriately**
   - README: Quick start, installation, basic usage
   - API docs: Endpoints, parameters, responses, examples
   - Architecture: System design, data flow, components
   - Guides: Step-by-step tutorials for common tasks

3. **Quality Standards**
   - Accurate — Reflects actual behavior
   - Complete — Covers all public interfaces
   - Concise — No unnecessary verbosity
   - Examples — Include working code samples
   - Up-to-date — Note version/date of documentation

4. **Format Consistently**
   - Use consistent heading hierarchy
   - Code blocks with appropriate syntax highlighting
   - Tables for structured data
   - Links between related sections

## Documentation Types
- **Inline comments**: Complex logic, non-obvious decisions
- **Docstrings**: Public functions, classes, modules
- **README**: Project overview, setup, usage
- **Architecture docs**: System design, patterns, decisions
- **API reference**: Endpoints, schemas, authentication
- **Changelog**: Version history, breaking changes

## Deliverable
Documentation that enables the target audience to understand and use the code without needing to read the implementation.
